## 环境基线与故障现象

**环境配置：**

- 操作系统：Ubuntu 22.04 LTS
- 当前内核：Linux 5.15.0-144-generic
- Kubernetes 版本：v1.28.15
- 容器运行时：containerd v1.7.27

**故障现象：**

- 节点在运行一段时间后，新调度的 Pod 无法启动，长期处于 `ContainerCreating` 状态。
- Kubelet 核心报错日志为：`unable to init seccomp: error loading seccomp filter into kernel: errno 524: unknown`。

------

## 2. 根本原因分析 (Root Cause Analysis)

经过深度排查，确认此故障并非 Kubernetes 或 Containerd 的业务层 Bug，而是触发了 **Linux 5.15 内核底层的 eBPF 内存泄漏与死锁缺陷**。具体诱发逻辑如下：

- **触发机制：** Kubernetes v1.28 默认开启了 `RuntimeDefault` 的 Seccomp（安全计算）限制。每次创建容器时，Containerd v1.7.x 都会向 Linux 内核注入一段 eBPF 程序作为安全过滤器。
- **内核缺陷：** Linux 5.15 版本的内核在处理 Cgroup v2 与 eBPF 程序的生命周期解耦时存在竞争缺陷（Race Condition）。当容器销毁时，内核无法正确将 eBPF 的引用计数清零，导致分配在内核态（`vmalloc` 区域）的 JIT 内存变成了永远无法回收的“僵尸内存”。
- **故障爆发：** 随着节点上 Pod 的频繁创建与销毁，僵尸内存不断累积，最终打满了内核的硬件架构安全保护上限（`bpf_jit_limit`），导致系统彻底拒绝为新的容器加载安全规则，触发 `errno 524` 报错。

## 升级执行指南 (Standard Operating Procedure)

请在业务低峰期的维护窗口（如凌晨）执行以下操作。

**第一步：节点隔离与驱逐（保障业务无损）** 在 Master 节点或中控机上执行，将目标节点设置为不可调度，并平滑驱逐其上的业务 Pod。

```bash
kubectl cordon k8s-master
kubectl drain k8s-master --ignore-daemonsets --delete-emptydir-data --force
```

**第二步：安装新版 HWE 内核** 登录目标节点，通过 apt 安装 Ubuntu 官方认证的硬件赋能内核。该过程非常安全，会自动处理依赖与 Grub 引导。

```bash
apt update
apt install --install-recommends linux-generic-hwe-22.04 -y
```

**第三步：重启服务器释放死锁内存** 安装完成后，必须重启物理机以卸载旧内核并清理泄漏的 eBPF 内存。

```bash
reboot
```

**第四步：验证升级结果** 机器重新上线后，登录节点执行以下命令确认内核版本已跨越 5.15：

Bash

```
uname -r
```

*预期输出示例：`6.5.0-xx-generic` 或 `6.8.0-xx-generic`。*

检查 eBPF 内存泄漏是否已清零：

```bash
cat /proc/vmallocinfo | grep bpf_jit | awk '{s+=$2} END {print s}'
```

*预期结果：数值应从之前的七亿多（约 745MB）断崖式降至几兆到十几兆。*

**第五步：恢复节点调度** 确认 Kubelet 运行正常且网络组件就绪后，恢复该节点的接客能力。

```bash
kubectl uncordon k8s-master
```
